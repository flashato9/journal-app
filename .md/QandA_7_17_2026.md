# Q&A

# General

## Why do I get `INSTALL_FAILED_UPDATE_INCOMPATIBLE` again even though I have a keystore at `~/.android/debug.keystore`?

Because the build never reads that file ‚Äî `android/app/build.gradle`'s `signingConfigs.debug` points at `android/app/debug.keystore` (the standard RN template key, SHA-256 `FA:C6:17:45‚Ä¶`), while the app on the phone is signed with a different cert (likely the EAS key you placed at `~/.android/debug.keystore`, SHA-256 `2C:8B:5B‚Ä¶`). Fix: copy the EAS keystore over `android/app/debug.keystore` and rebuild (keeps app data), or `adb uninstall com.relentless.memory_journal` (wipes data).

## What did "commit the keystore change so it sticks" mean?

The `Copy-Item` fix is only a local file change ‚Äî `prebuild --clean`, deleting `android/`, or a fresh clone restores the template key and brings the error back; putting the EAS keystore in git makes it permanent. Since `android/` is gitignored here, that requires `git add -f android/app/debug.keystore`.

## Why did the build fail with "No key with alias 'androiddebugkey' found" after copying the EAS keystore?

The EAS keystore's key entry has a generated alias (`74e2f989‚Ä¶`), but build.gradle signs with alias `androiddebugkey`. Fix: `keytool -changealias` to rename the alias to `androiddebugkey` (and `keytool -keypasswd` if the key password isn't `android`).

## What does `keytool -changealias` do?

Renames a key entry inside a keystore (a keystore is a small database of keys, each stored under an alias/name). The key and certificate are untouched, so the signature stays identical.

## Why isn't the key password `android` when I already changed the keystore password to `android`?

A JKS keystore has two separate passwords: the store password (opens the file ‚Äî the one changed to `android`) and a per-key password protecting each key entry, which is still the EAS-generated one. Get it from `eas credentials` or expo.dev, then align it with `keytool -keypasswd`.

## What does the build error "Cannot recover key" mean?

Gradle found the key entry (alias is right now) but the key password it's using (`android` from build.gradle) doesn't match the entry's actual password ‚Äî fixed by `keytool -keypasswd` setting the key password to `android`.

# .env.signing

## What are a keystore and an alias?

A keystore is a password-protected vault file holding private keys; an alias is the name of one entry inside it. Like a safe with named drawers: the store password opens the safe, each drawer (alias) holds one key + certificate and can have its own key password ‚Äî Gradle's signing config is just those four coordinates (file, store password, alias, key password).

# services/locationService.ts

## Why do I get two similar "Journal is tracking your walk" notifications on login?

They're two separate notifications: the Android-mandated foreground-service notification (static body, posted natively by expo-location) and the app's own live-status notification (posted via expo-notifications, updated in place each cycle). Both share the same title, so they read as duplicates ‚Äî a side effect of moving live updates out of the (previously patched) service notification into a separate one.

## Why does tracking start but not keep checking periodically?

The background task only fires when the OS delivers a location ‚Äî `timeInterval` is a request, not a timer. On Xiaomi phones the usual causes are MIUI battery restrictions freezing the app (set "No restrictions" + Autostart), no GPS fix indoors, or the app being swiped away (lock it in recents). Verify with `adb logcat -s ReactNativeJS:V` (task banners) and `adb shell dumpsys activity services com.relentless.memory_journal` (service alive?).

## How does the background location task actually work (plain terms)?

The app never wakes itself up ‚Äî it asks Android "deliver my location every ~10s" and its handler code runs only when Android actually delivers one (like someone who answers the door only when mail arrives). If deliveries stop (battery saver freeze, no GPS fix, app killed), the handler is fine but never runs.

## Is the task/permission setup correct in code, and why might the task still not fire?

Manifest permissions, the `foregroundServiceType="location"` service, module-scope `defineTask`, and runtime permission checks are all correct. The gap: on Android 12+ "Approximate" location still returns `granted`, but coarse-only apps get throttled to ~1 update per 20 min ‚Äî check the "Use precise location" toggle in app permission settings; also verify the DB-configured fetchFrequency in the start log and whether GPS has a fix.

## Bottom line ‚Äî what was actually wrong with location tracking (no logs)?

Nothing in the code. The phone wasn't getting a GPS fix indoors (dumpsys: 4 satellites ~13 dB-Hz, last real fix 11 days old), and with `HIGH_ACCURACY` (GPS-primary) no fix means no location delivered, so the task never ran and produced no logs ‚Äî GPS starvation shows up as silence, not an error. It resolved once a real fix came through (task then fired even at High). Two unrelated issues appeared while debugging: "Unable to load script" (self-inflicted ‚Äî Metro stopped + `adb reverse` tunnel removed; debug builds load JS from Metro) and a hung `expo run:android` (the recurring zombie Kotlin/Gradle daemon).

## Why didn't the phone log anything about the weak GPS signal?

Because weak signal is an absence, not an error. All location logs live inside the task handler, which Android only calls when it has a completed fix ‚Äî no fix means the handler never runs and there's nothing to log. expo-location's callback only exposes finished fixes plus an `error` slot for permission/provider failures, never signal quality (raw satellite count/C¬∑N0 needs a `GnssStatus.Callback` on the low-level LocationManager, which expo-location doesn't surface). The satellite detail was only visible via `adb shell dumpsys location`, the OS's own diagnostic ‚Äî the app has no API access to it. So a starved GPS reads as silence (missing `üîµ` banners), not an error.

## Registration says ‚úì ACTIVE but no task banners fire ‚Äî why?

Confirmed via dumpsys it's GPS signal, not code/patch/battery: `dumpsys activity services` shows `LocationTaskService isForeground=true` (service alive) and `dumpsys location` shows the request registered as `HIGH_ACCURACY, allowed=true` (precise, not throttled) ‚Äî but `last location=[gps ‚Ä¶ et=+11d9h‚Ä¶, satellites=4, meanCn0=13]`, i.e. last real fix 11 days old with indoor-level signal (~13 dB-Hz vs the ~30‚Äì45 needed). The task only fires when a fix is delivered, so no sky view = no banners. Go outside/near a window, or switch `Accuracy.High`‚Üí`Balanced` to allow Wi-Fi/cell fixes indoors.

## Was the deleted expo-location patch needed for location tracking / the task to fire?

No. The patch only added `updateForegroundServiceNotificationAsync` (update the tracking notification's text in place, for the live-status feature) and `.setOngoing(true)` (cosmetic: non-dismissible notification on Android ‚â§13, ignored on 14+). The foreground service and location delivery come from the stock `foregroundService` option passed to `startLocationUpdatesAsync`, which predates the patch ‚Äî so removing the patch is fine and does not stop the task banner logs.

## Why am I not seeing any location-tracking logs at all?

The code only logs at two moments: the `startLocationTracking` burst (login or saving location settings) and the task banner when Android actually delivers a fix ‚Äî between fixes it is silent by design (the 15s status poll logs nothing). So total silence means either you're watching the wrong stream (Metro only shows logs while the dev client is connected; logcat always has them) or the provider is delivering no fixes, which was the earlier diagnosis.

## Shouldn't I be seeing a log when I press the location icon?

No ‚Äî the header icon's `handleLocationIconPress` only shows an `Alert` ("Online"/"Offline") and contains no `console.log`; the status refresh it reads from (`isLocationTrackingActive`) is also silent on success. Pressing the icon triggers no location code, so no terminal output is expected.

# app.json

## What's the full picture of how location tracking works and where it's failing?

Login ‚Üí permission checks ‚Üí `startLocationUpdatesAsync` registers a request with Android's fused location provider and starts the foreground service; from then on the app is passive ‚Äî the JS pipeline (record ‚Üí distance-from-memory ‚Üí rest detection ‚Üí break notification) only runs when the provider emits a fix. The startup notifications prove registration succeeded, so the provider is withholding fixes: approximate-only permission, no GPS fix, OEM power manager, or a large DB fetchFrequency. `adb shell dumpsys location` shows the registered request and last-delivery times to pinpoint which.

## Do the `permissions` in app.json affect the installed app directly?

No ‚Äî they're only copied into `android/app/src/main/AndroidManifest.xml` at `expo prebuild` time, and they only _declare_ what the app may request. The actual grant state (including precise vs approximate) lives on the phone.
